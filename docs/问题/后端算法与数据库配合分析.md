## 一、数据模型层

| 表 | 用途 | 设计 |
|---|------|------|
| `error_analysis_tasks` | 存储任务（算法、配置、状态、元数据） | ✅ 正确，`result_metadata` JSON 字段支持各算法存储不同结果 |

| `track_interpolated_points` | 插值点 | ⚠️ 同上 |
| `flight_tracks_raw` | 原始轨迹 | ✅ 所有算法都读这个表 |

## 二、各算法与数据库配合情况

### 2.1 算法参数对比

| 算法 | 模式 | 核心参数 | 特有参数 |
|------|------|---------|---------|
| **gradient_descent** | 多源参考 | grid_resolution, time_window, match_distance_threshold, optimization_steps, range_optimization_steps, cost_weights | (基础算法，无特有参数) |
| **ransac** | 多源参考 | 继承上述 + optimization_steps, range_optimization_steps, cost_weights | **residual_threshold**, **min_samples**, **max_iterations**, **outlier_ratio_threshold** |
| **weighted_lstsq** | 多源参考 | 继承上述 + optimization_steps, range_optimization_steps, cost_weights | **weighting_method**, **outlier_removal** |
| **kalman** | 单源盲测 | min_track_points | **process_noise**, **measurement_noise**, **initial_uncertainty** |
| **particle_filter** | 单源盲测 | min_track_points | **num_particles**, **process_noise_std**, **measurement_noise_std**, **resampling_method**, **effective_particle_threshold** |
| **spline** | 单源盲测 | min_track_points | **smoothing_factor**, **spline_degree**, **interpolate**, **interpolation_density** |

### 2.2 各算法工作原理

| 算法 | 原理 |
|------|------|
| **gradient_descent** | 多雷达航迹匹配后，用梯度下降迭代优化误差函数，求解各站系统误差（方位角/距离/俯仰角） |
| **ransac** | 在 gradient_descent 基础上，对每个匹配组做 RANSAC 回归，区分内点和离群点，统计各站离群率，识别故障站 |
| **weighted_lstsq** | 计算每站与组质心的偏差作为"误差"，按 inverse_variance/uniform/match_count 之一加权融合，输出最优轨迹和各站误差 |
| **kalman** | 单站数据用 6状态(位置+速度) KalmanFilter 预测-修正，适合高斯噪声，输出平滑轨迹 |
| **particle_filter** | N个粒子采样→预测→根据观测似然加权→重采样，适合非线性非高斯，输出概率分布轨迹 |
| **spline** | 对经纬度高度分别用 UnivariateSpline 拟合，可选插值增加密度，输出平滑曲线 |

### 2.3 数据库写入情况

| 算法 | 输入数据 | 写入数据库 | 问题 |
|------|---------|-----------|------|
| **gradient_descent** | `radar_station_ids`, `track_ids` | `track_segments`, `match_groups`, `track_interpolated_points`, `error_results` | ✅ 正常 |
| **ransac** | `radar_station_ids`, `track_ids` | 同上 + `error_results` | ✅ 正常 |
| **weighted_lstsq** | `radar_station_ids`, `track_ids` | 同上 + `error_results` | ✅ 正常，`result_metadata` 存储 `fused_trajectory` |
| **kalman** | 单站数据 | 只写 `error_results` | ✅ 正常 |
| **particle_filter** | 单站数据 | 只写 `error_results` | ✅ 正常 |
| **spline** | 单站数据 | 只写 `error_results` | ✅ 正常 |

## 三、潜在问题分析

### 问题 1：单源盲测算法不写中间数据

`kalman`/`particle_filter`/`spline` 这三个算法**不经过** MRRA 流水线（提取→插值→匹配），所以不会产生：
- `track_segments`
- `match_groups`
- `track_interpolated_points`

**影响**：任务详情页的"航迹段"和"匹配组"标签页对单源盲测算法**没有数据可显示**。

### 问题 2：单源盲测"误差"定义不同

这三个算法输出的 `azimuth_error`、`range_error`、`elevation_error` **不是传统意义上的系统误差**，而是：
- 平滑前后偏差的 RMSE

这是一个语义问题。传统误差分析是"各站相对于真值的偏移"，而这些单源算法比较的是"平滑后 vs 原始观测"。

### 问题 3：AlgorithmFactory 的 config 解析

```python
# factory.py 第 78-82 行
if config_class:
    config = config_class(**config_dict)  # 可能传了多余字段
else:
    config = config_dict
```

对于 `ransac`/`weighted_lstsq` 等，如果前端传来 `grid_resolution`、`time_window` 等 MRRA 参数，但 `RansacAlgorithmConfig` 也有这些字段（继承或直接定义），解析是 OK 的。

但是如果前端传来 `process_noise`（kalman 的参数）给 `ransac`，Pydantic 会**忽略未知字段**，不会报错。

### 问题 4：俯仰角误差语义

检查代码：
- `KalmanAlgorithm.supports_elevation()` → 返回 `True`
- `ParticleFilterAlgorithm.supports_elevation()` → 返回 `True`
- `SplineAlgorithm.supports_elevation()` → 返回 `True`

但这些算法的"俯仰角误差"其实是 altitude 的 RMSE，不是真正的俯仰角误差。

## 四、后端设计评价

| 方面 | 评价 |
|------|------|
| **数据模型设计** | ✅ 通用化设计，`error_results` 表足够通用 |
| **算法框架** | ✅ 良好的扩展性，Registry + Factory 模式 |
| **算法实现** | ✅ 各算法逻辑完整，接口统一 |
| **多源 vs 单源分离** | ✅ 多源走 MRRA 流水线，单源直接读 `flight_tracks_raw` |
| **`result_metadata` 存储** | ✅ JSON 字段灵活存储算法特定结果 |
| **与数据库配合** | ⚠️ 单源盲测算法不产生中间数据（航迹段/匹配组），前端详情页需要适配 |

## 五、结论

后端 + 数据库配合**整体是完善的**，但有一个关键问题：

**任务详情页**（`ErrorAnalysisHistory.vue`）假设所有任务都有 `track_segments` 和 `match_groups`，但单源盲测算法（kalman/particle_filter/spline）不会产生这些数据。前端需要根据 `algorithm_name` 决定显示哪些标签页。

## 六、前端重构方向

### 6.1 配置页面重构

| 原设计 | 问题 | 重构方向 |
|--------|------|---------|
| 共用一个 `config` 对象 | 单源盲测算法收到不认识的参数（如 grid_resolution、optimization_steps） | 按算法类型加载独立配置 Schema |
| `ErrorAnalysisConfig` 类型与 `MrraConfig` 绑定 | 只适合多源参考算法，不适合单源盲测 | 各算法独立配置组件 |

### 6.2 任务详情页重构

| 原设计 | 问题 | 重构方向 |
|--------|------|---------|
| 所有任务显示相同标签页 | 单源盲测没有航迹段和匹配组数据 | 根据 `algorithm_name` 动态显示标签 |
| 多源参考和单源盲测共用结果展示 | "系统误差" vs "平滑效果" 完全不同 | 按模式分类展示 |

### 6.3 算法分类

```
多源参考模式 (gradient_descent / ransac / weighted_lstsq)
  ├── 需要多雷达站数据
  ├── 产生中间数据（航迹段/匹配组/插值点）
  └── 结果：系统误差 + 融合轨迹

单源盲测模式 (kalman / particle_filter / spline)
  ├── 单站数据即可
  ├── 不产生中间数据
  └── 结果：平滑效果对比（原始 vs 平滑后偏差）
```

## 七、数据库表结构参考

### error_analysis_tasks
```sql
- id, task_id, radar_station_ids (JSON), track_ids (JSON), user_id
- algorithm_name, config (JSON), status, progress, error_message
- result_metadata (JSON) -- 存储算法特定结果（融合轨迹/离群率等）
- created_at, started_at, completed_at
```

### error_results
```sql
- id, task_id, station_id
- azimuth_error, range_error, elevation_error
- match_count, confidence, iterations, final_cost
- created_at
```

### track_segments / match_groups / track_interpolated_points
```sql
-- 仅多源参考算法会写入
- track_segments: task_id, segment_id, station_id, track_id, start_time, end_time, point_count
- match_groups: task_id, group_id, match_time, match_points (JSON), point_count, avg_distance, max_distance, variance
- track_interpolated_points: task_id, segment_id, station_id, track_id, time_seconds, timestamp, longitude, latitude, altitude, is_original
```
